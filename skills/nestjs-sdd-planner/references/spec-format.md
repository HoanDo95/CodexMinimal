# Spec Format

A good CodexMinimal spec is concise, implementation-ready, and bounded.

Suggested section order:

1. Scope
2. Non-goals
3. Affected modules
4. Protected-file impact
5. API contract
6. DTO design
7. Service behavior
8. Persistence behavior
9. Validation and errors
10. Acceptance criteria
11. Test plan
12. Implementation handoff notes
13. Open questions or assumptions

Rules:

- no placeholders
- no hidden assumptions
- no vague wording like "handle appropriately"
- use exact module or file names when known
